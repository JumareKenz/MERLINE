# Retrieval-Augmented Generation (RAG) Architecture

## 1. Overview

Merline's RAG system grounds LLM responses in verified knowledge sources, reducing hallucination and ensuring outputs are traceable to source material. Every AI-generated claim must cite its sources.

## 2. Knowledge Sources

### 2.2 Ingestion Pipeline

```
Document Upload
    │
    ▼
[1] Pre-processing
    ├── Format detection (PDF, DOCX, HTML, Markdown, plain text)
    ├── PII scanning & redaction (if configured)
    └── Language detection
    │
    ▼
[2] Chunking
    ├── Strategy: Semantic sentence-based chunking
    ├── Chunk size: 512-1024 tokens (configurable per source type)
    ├── Overlap: 10-20% (for context continuity)
    └── Preserve: Headers, list structures, tables
    │
    ▼
[3] Metadata Extraction
    ├── Source ID, document title, URL
    ├── Document type, creation date, version
    ├── Tenant ID (organization_id)
    ├── Entity tags (study, indicator, project references)
    └── Summary (generated by LLM on ingest)
    │
    ▼
[4] Embedding Generation
    ├── Model: Voyage-3 (primary), BGE-M3 (fallback/self-hosted)
    ├── Dimension: 1024 (Voyage) / 1024 (BGE)
    ├── Batch: 100 chunks per batch
    └── Async via queue
    │
    ▼
[5] Storage
    └── pgvector (PostgreSQL extension)
        ├── Table: document_chunks
        ├── Columns: id, chunk_text, embedding, metadata (JSONB), tenant_id, created_at
        ├── Index: HNSW (cosine distance)
        └── Partitioned: BY RANGE (created_at) with tenant_id as partition key
```

### 2.3 Chunking Strategies by Source Type

| Source Type | Chunk Strategy | Size | Overlap |
|------------|---------------|------|---------|
| MERL Frameworks | Section-based (by heading) | 1024 tokens | 15% |
| SOPs | Section-based | 768 tokens | 10% |
| Reports | Semantic paragraph + heading | 512 tokens | 10% |
| Questionnaires | Per-question (preserve structure) | 256 tokens | 0% |
| Lessons Learned | Full entry (usually <512 tokens) | Full | 0% |
| Academic Papers | Section-based (abstract, intro, etc.) | 512 tokens | 10% |
| Tables | Row-based with column headers | 256 tokens | 0% |

## 3. Retrieval Pipeline

### 3.1 Query Processing

```
User Query
    │
    ├── 1. Query Rewriting (LLM)
    │     ├── Expand acronyms, clarify ambiguous terms
    │     └── Generate 1-3 alternative phrasings
    │
    ├── 2. Query Embedding
    │     └── Same model as document embeddings (Voyage-3)
    │
    └── 3. Hybrid Search
          ├── Vector similarity (cosine, top K=50)
          ├── Full-text search (PostgreSQL tsvector, top K=50)
          ├── Metadata filter (tenant, source type, date range)
          └── Weighted fusion: α = 0.7 (vector) + 0.3 (FTS)
    
    ▼
[Re-ranking]
    ├── Cross-encoder model (BGE-reranker-v2 / Cohere rerank)
    ├── Top 50 → top 10 (max 3 per source type for diversity)
    └── Score = relevance to query
    
    ▼
[Context Compression]
    ├── Extractive: Keep top 10 chunks sorted by score
    ├── Compressive: Summarize redundant chunks (if budget allows)
    └── Final context: max 4096 tokens (dynamically adjusted per model context window)
```

### 3.2 Search Parameters by Capability

| Capability | K (initial) | K (final) | Filter | Weight (α) | Rerank |
|-----------|-------------|-----------|--------|------------|--------|
| **RAG Q&A** | 50 | 10 | Tenant + source type | 0.7 | Yes |
| **Report Drafting** | 100 | 20 | Tenant + date range | 0.6 | Yes |
| **Indicator Generation** | 30 | 8 | Indicator library + frameworks | 0.8 | Yes |
| **Questionnaire Design** | 50 | 12 | Questionnaires + indicators | 0.7 | Yes |
| **ToC/LogFrame** | 80 | 15 | Past designs + frameworks | 0.6 | Yes |
| **Knowledge Assistant** | 50 | 10 | All sources | 0.7 | Yes |
| **Data Quality Check** | 20 | 5 | Validation rules + guidelines | 0.9 | No |

### 3.3 Feedback Loop for Retrieval

```
User rates response
    │
    ├── Positive → Store query + retrieved chunks as "good example"
    ├── Negative → Analyze reasons
    │     ├── "Missing information" → Tune recall (increase K, reduce α_vector)
    │     ├── "Irrelevant" → Tune precision (increase rerank score threshold)
    │     ├── "Wrong source" → Check metadata filtering
    │     └── "Outdated" → Adjust date recency boosting
    │
    └── Periodic: Retrain reranker on curated pairs (quarterly)
```

## 4. Hybrid Search Implementation

### 4.1 PostgreSQL + pgvector

```sql
-- Schema
CREATE TABLE document_chunks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id UUID NOT NULL REFERENCES documents(id),
    tenant_id UUID NOT NULL,
    chunk_index INTEGER NOT NULL,
    chunk_text TEXT NOT NULL,
    embedding vector(1024),
    metadata JSONB NOT NULL DEFAULT '{}',
    fts tsvector GENERATED ALWAYS AS (to_tsvector('english', chunk_text)) STORED,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    
    -- Full-text search index
    CONSTRAINT fk_document FOREIGN KEY (document_id) REFERENCES documents(id)
);

-- Indexes
CREATE INDEX idx_chunks_tenant ON document_chunks(tenant_id);
CREATE INDEX idx_chunks_fts ON document_chunks USING GIN(fts);
CREATE INDEX idx_chunks_embedding ON document_chunks 
    USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 200);
CREATE INDEX idx_chunks_metadata ON document_chunks USING GIN(metadata jsonb_path_ops);

-- Hybrid search query
SELECT 
    id, chunk_text, metadata,
    (1 - (embedding <=> $query_embedding)) AS vector_score,
    ts_rank(fts, plainto_tsquery('english', $query_text)) AS fts_score
FROM document_chunks
WHERE tenant_id = $tenant_id
    AND metadata @> $metadata_filter
ORDER BY 
    (0.7 * (1 - (embedding <=> $query_embedding)) + 
     0.3 * ts_rank(fts, plainto_tsquery('english', $query_text))) DESC
LIMIT $k;
```

### 4.2 Performance Targets

| Metric | Target | At Volume |
|--------|--------|-----------|
| Query embedding | <200ms | 100 QPS |
| Vector search (50K chunks) | <50ms | — |
| Vector search (1M chunks) | <100ms | HNSW index |
| Hybrid search | <200ms | — |
| Re-ranking (50 → 10) | <300ms | GPU |
| Total retrieval | <1s | — |
| Index refresh | <1s | Real-time (per insert) |

## 5. Citation System

### 5.1 Citation Format

```
Every AI response includes inline citations:

"The OECD DAC evaluation criteria include relevance, coherence, 
efficiency, effectiveness, impact, and sustainability[1]."

[1] OECD DAC Evaluation Criteria, OECD Publishing, 2019, Page 12
    Source: MERL Frameworks (knowledge_base_id: 12345, chunk_id: abc-123)
```

### 5.2 Citation Validation

| Check | Method |
|-------|--------|
| Citation exists | Verify chunk_id is in retrieved set |
| Claim matches source | Semantic similarity check between claim and chunk |
| Source is authoritative | Check source_type ranking (official > derived) |
| Source is up-to-date | Check document date vs current date |
| Multiple sources | Cross-validate high-claims against 2+ sources |

## 6. Tenant Isolation & Security

| Mechanism | Implementation |
|-----------|---------------|
| **Data isolation** | All queries filter by tenant_id at database level |
| **Metadata filtering** | tenant_id in metadata JSONB, indexed |
| **Cross-tenant leakage** | Enforced by RLS (Row-Level Security) on document_chunks |
| **PII redaction** | On ingest: scan for PII patterns, redact or exclude chunk |
| **Audit trail** | Every retrieval logged with query, results, user, timestamp |
| **Retention** | Documents retained per tenant data retention policy |
| **Deletion** | Tenant deletion cascades to all chunks and embeddings |
